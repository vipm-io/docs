# Lock Files (`vipm.lock`)

The `vipm lock` command initializes a `vipm.lock` file in your workspace, right next to your `vipm.toml` file. The lock file **contains an exact list of all the packages your project depends on** (including transitive dependencies), so that this state can be reproduced for builds, testing, and other developers.

`vipm.lock` also records the cryptographic hashes (checksums) of the package files, so they can be verified bit-for-bit, and each package's origin location, so packages can be downloaded in the future and their provenance can be tracked for compliance and other purposes — see [Software Bill of Materials](../sbom/index.md).

!!! note "Generated by vipm"
    `vipm.lock` is generated by vipm and is not intended for manual editing.

## Why lock files are useful

- **One exact answer to "which versions does this project use?"** `vipm.toml` declares what your project needs; `vipm.lock` records what that resolves to — exact versions for every package, including transitives — so teammates, CI, and future builds all reference the same state.
- **Dependency changes show up in code review.** The lock file is committed and deterministically ordered, so a version bump — or a new transitive dependency you didn't know you were pulling in — appears as a readable diff in the pull request that caused it.
- **Drift gets caught instead of discovered.** `vipm lock --check` fails CI when the lock no longer matches `vipm.toml`, and `vipm build` / `vipm sbom` refuse to produce output when installed packages disagree with the project's declared state.
- **Packages can be verified and traced.** Recorded checksums let a downloaded package file be verified bit-for-bit, and recorded origins say where every package came from — the raw material for SBOMs and compliance audits.

## When the lock file is created and updated

Lock files are **opt-in**: no `vipm.lock` exists until you create one with `vipm lock`. If your workflow depends on a lock file, run `vipm lock` once and commit the result; `vipm install` keeps it up to date from then on.

| Command | Effect on `vipm.lock` |
|---------|-----------------------|
| `vipm lock` | **Creates** the file if missing; updates it otherwise. The only command that creates a lock file. |
| `vipm install` | Refreshes an existing lock after installing; does **not** create one. |
| `vipm add` / `vipm remove` | Never touch the lock — they edit `vipm.toml` only. |
| `vipm add --install` | Like `vipm install`: refreshes an existing lock; does not create one. |

`vipm lock` requires [**VIPM Community** or higher](../vipm-editions-comparison.md#available-editions).

!!! note "Behavior change in 2026 Q3"
    Earlier previews created `vipm.lock` automatically during `vipm install` and `vipm add`. As of **VIPM 2026 Q3**, creation is opt-in: only `vipm lock` creates the file. See the [2026 Q3 release notes](../release-notes/2026.3.md#vipmlock-creation-is-now-opt-in).

## Generating a lock file

```bash
# Create or update vipm.lock from vipm.toml
vipm lock

# Keep going when some package specs can't be fetched
# (writes what it can resolve; warns instead of erroring)
vipm lock --best-effort
```

- `vipm lock` resolves the full dependency graph declared in `vipm.toml` and writes `vipm.lock` next to it. Run it from anywhere inside the project — it searches parent directories for `vipm.toml`.
- The lock **always includes dev-dependencies**, matching the convention of other lock-file-based package managers; dev packages are marked `group = "dev"`. There is no flag to exclude them.
- Output is deterministic: regenerating with the same inputs produces an identical file, so lock-file diffs in version control stay meaningful.
- If nothing changed, the file is left untouched.

## Checking lock file sync (in CI)

```bash
# Verify vipm.lock matches vipm.toml without writing anything
vipm lock --check
```

- **In sync** → exit code `0` and `vipm.lock is in sync with vipm.toml`.
- **Out of sync** → an error with exit code `2`, telling you to run `vipm lock`.
- **No lock file** → an error with exit code `2` (the check requires an existing `vipm.lock`).

See [Exit Codes](../cli/command-reference.md#exit-codes) for the full catalog.

## What the lock file records

For every package your project depends on — directly or transitively — the lock file records:

- **The exact resolved version**, so every machine builds and tests against the same dependency set.
- **Where it comes from** — the VIPM repository or NIPM feed it can be obtained from (with direct download locations for NIPM packages), so the same packages can be fetched again later and their origin is on record.
- **An integrity checksum**, so a downloaded package file can be verified bit-for-bit against what was locked.
- **Its place in the dependency graph** — what it depends on, and whether it's a dev-dependency.

The file also records your project's LabVIEW requirements, when the lock was last updated, and a schema `version`/`revision` pair so tools know how to read it.

`vipm.lock` is human-readable TOML — a trimmed example:

```toml
# This file is automatically generated by vipm.
# It is not intended for manual editing.

version = 2
revision = 0
updated-at = "2026-03-27T21:56:46Z"
requires-labview = "2024"

[[nipm.feed]]
name = "ni-labview-2026-core-en-2026 Q1-released"
uri = "https://download.ni.com/support/nipkg/products/ni-l/ni-labview-2026/26.1/released"

[[vipm.repo]]
name = "VIPM Community"
uri = "http://www.jkisoft.com/packages"

[[package]]
name = "jki_lib_state_machine"
version = "2018.0.7.45"
manager = "vipm"
dependencies = ["jki_rsc_toolkits_palette"]
resolution-source = "repo"

[[package.source]]
repo = "VIPM Community"
checksum = "sha256:a1b2c3d4..."

[[package]]
name = "ni-auth"
version = "26.1.0.49284-0+f132"
manager = "nipm"
dependencies = ["ni-ceip", "ni-mdfsupport", "ni-metauninstaller"]

[[package.source]]
feed = "ni-labview-2026-core-en-2026 Q1-released"
architecture = "windows_x64"
checksum = "sha256:228b8839..."
uri = "https://download.ni.com/support/nipkg/pool/ni-a/ni-auth/26.1.0/ni-auth_26.1.0.49284-0+f132_windows_x64.nipkg"

# ... plus one [[package]] entry for every transitive dependency
```

Key fields on each `[[package]]` entry:

- `name` / `version` — the exact resolved package identity
- `manager` — which package manager owns the package (`"vipm"` or `"nipm"`)
- `dependencies` — names of the packages it depends on
- `group = "dev"` — present on dev-dependencies; omitted otherwise
- `resolution-source` — where the version was resolved from (`"repo"` or `"installed"`)
- `[[package.source]]` — where the package can be obtained: a `repo` (VIPM) or `feed` (NIPM) reference into the registry tables, with an optional integrity `checksum`, and for NIPM packages the target `architecture` and a direct download `uri`

Lock files written by earlier previews (schema `version = 1`) are still read; the next `vipm lock` or `vipm install` rewrites the file in the current format.

## How the lock file is used

- **Drift checking** — [`vipm lock --check`](#checking-lock-file-sync-in-ci) fails CI when `vipm.toml` and `vipm.lock` disagree.
- **Dependency-state verification** — `vipm build` and `vipm sbom` verify that installed packages agree with the project's declared state (`vipm.toml` / `vipm.lock`) before producing output. See [Dependency-State Verification](../cli/command-reference.md#dependency-state-verification).
- **SBOM generation** — when a fresh `vipm.lock` is present, `vipm sbom` takes resolved versions from it. See the [SBOM Output Reference](../sbom/output-reference.md).
- `vipm install` installs directly from an in-sync `vipm.lock` when one exists (the locked VIPM closure), falling back to resolving from `vipm.toml` when the lock is missing, out of sync, or when `--upgrade` is passed; it refreshes an existing lock afterward. Lock-based install selection currently applies to VIPM packages; NIPM packages still install via manifest resolution.

## Best practices

- **Commit `vipm.lock`** to version control once you opt in, so drift checks and audits have a stable reference.
- **Run `vipm lock --check` in CI** to catch out-of-sync lock files early.
- **Re-run `vipm lock`** after editing dependencies in `vipm.toml`.
- **Don't edit the file by hand** — regenerate it with `vipm lock`.
